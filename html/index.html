Automagic

 This is a full-stack application built on the MEAN stack. In short, it automates several repetitive tasks at my current job. It offers an interface where employees can gather data from the internet, compare that with information in a database (which is compiled and maintained by the application and by the users), and then presents the results back to the user in a directly useable format.

 At first, I began writing the code for this project just to see if it was possible, but as the project grew, I knew I had to share it. I built a prototype to show to my supervisors, and they told me to go ahead with the project. After completing a full version version, I was able to coordinate a beta test with a handful of other employees. With positive feedback from the testers, I had the application approved for company-wide use by management. Now, I am happy to say it used as a part of everyday workflow within the company.

The full story...

I will try not to bore you too much with the details, but I currently work as a legal writer for an immigration law firm that helps scientists gain permanent visas to the United States. One thing we have to do is get information about every journal where they have published articles, how often they have been cited, etc. We would put this together for each application, but the information was never organized or stored anywhere, so each time we would start fresh. I found myself looking up the same information over and over again, and it bothered me to think that there might be multiple people doing almost the exact same task. Everyone knew the system was tiresome and inefficient, but no one had a solution.

After learning a little programming, I realized I could use those skills to make a better system. I began by writing a web crawler that used regular expressions to pull all the relevant information off of a Google Scholar profile. Next, I built a second web crawler that pulled the top 10 journals from each Google Scholar category (sorry, Google) and stored that information in a Mongo database. Then, I wrote a script to parse a spreadsheet from Thomson Reuters showing the "Impact Factor" of every journal, and combined that with the other information in the database. I also wrote code that compares how much each article has been cited with charts (also from Thomson Reuters) to tell us if the article is considered "highly cited."

In a perfect world, this would have been very simple, but I found that there were so many inconsistencies in the data that the majority of the work I put into the app was focused on creating the interface that allows users to gather, edit and modify the data as needed. For instance, clients often list journals by abbreviation (or just plain spell them wrong), and Google Scholar and Thomson Reuters may differ in how they name the same journal. Users therefore need to be able to monitor for such inconsistencies and make changes as needed. At the same time, the whole purpose of the application is to save time, so I had to make the process as seamless as possible. While I know the user interface is far from attractive, I do believe it is intuitive, efficient and very effective at what it does. I think this was my favorite part of the project and I would love to do more work like this in the future.

Another challenge I had to overcome is that we have to provide "evidence" of all information submitted to the immigration office. To accomplish this, I made each entry on the database store a link to where that information came from. After obtaining the list of journal rankings, the user clicks a button to download a zip file of pdf copies for each of these webpages. A lot of the heavy lifting is done by npm packages. I use a crawler (tinyreq) to grab the html source of each page, this is then fed to a phantom.js wrapper (html-to-pdf) that (you guessed it) converts the html to a pdf. These are then complied into a zip file using jszip. Even so, coordinating all these modules to work together felt like a big feat for me, and I am really happy with the outcome.

Finally, I had to build a "back-end" for the application where users could update and maintain the database, as journal rankings and citation percentiles are constantly being updated. This way, the application can remain in use in perpetuum even after I leave the company.

Using my app to gather this information takes between 5-10 minutes from start to finish. Before, it took half an hour minimum, and up to an hour or more for especially prolific clients. By my estimates, I am saving the company close to 40 hours per week. It's like I've given them a full-time employee who works for free. I can't help but think that this is really awesome.

Personally, even though it might not mean much to an outsider, I am very proud of my work. I identified a problem, came up with a solution, wrote all the code, pitched the app to the company, oversaw testing, and have even trained people in its use. Back when I first started, I knew almost nothing about back-end programing and asynchronous code was a total mystery to me. Along the way, I had to teach myself everything that went into this. I was doing the work all on my own initiative, putting in long hours at home nights and weekends, and had to stay confident that I was making something worthwhile. I am thrilled to have built something that people are using every day, that improves productivity and saves employees from mindless toil. I can't wait to see what I can do while working

Javascript Rogue-Like

This is a "fun" side-project I have been working on when I want a "break" from my other coding projects. The original code was written following a tutorial, but I've expanded massively from where that left off. It uses the ROT.js library by Ondrej Zara.

I am collaborating on this project with an old friend who doesn't really know how to code, but loves rogue-likes and has tons of great ideas. So, I build the functions and put everything together, and he does content development. This has forced me to write cleaner code that offers simple abstractions he can deal with. It has been an interesting challenge for me, and has done a lot to improve my command of JS.
